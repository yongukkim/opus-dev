# OPUS Security Coding Standards (ISO 27001 Based)

본 문서는 OPUS 프로젝트의 보안 무결성과 글로벌/일본의 개인정보보호 규정을 준수하기 위해 AI 에이전트가 준수해야 할 8대 보안 코딩 규칙을 정의한다.

지침: 모든 코드 작성 시 프로젝트 루트의 SECURITY_GOVERNANCE.md에 명시된 ISO 27001 보안 코딩 규칙을 엄격히 준수하라. 특히 일본 개인정보보호법(APPI)과 관련된 데이터 처리 로직 작성 시 보안 리뷰 주석을 반드시 포함하라.

## 개발·배포 환경 (로컬 1대 → 클라우드 분리)
* **로컬 개발:** **단일 노트북(또는 동급 개발 머신 1대)** 에서 진행한다. 맥미니 등 **별도 기기를 개발 환경에 연결해 두는 전제**는 두지 않는다. DB는 같은 머신의 Docker/로컬 Postgres 등으로 기동하거나, 환경변수로 **클라우드 스테이징 DB**를 가리키도록 전환할 수 있게 둔다.
* **클라우드·운영:** 스테이징/프로덕션에서는 **WEB·WAS(API)·DB** 등을 **서버(역할) 단위로 분리**해 배포한다. 상세·경계는 `SECURITY_GOVERNANCE.md` §3.1과 맞춘다.

## 일본 기업 인수(M&A) 적합성
* **목표:** 이후 **일본 기업에 사업·자산 매각**이나 **일본 법인 중심 운영**으로 이행할 때, 기술·컴플라이언스 **실사(Due Diligence)** 가 원활하도록 처음부터 설계한다.
* **실행:** **SECURITY_GOVERNANCE.md §0**을 따른다. 요약하면, (1) 보안·PII 로직의 **ISO/APPI 추적 주석(설명은 한·일·영 삼어, §5)**, (2) 데이터·Chronicle·PII의 **논리적 분리**, (3) **라이선스가 명확한** 의존성만 사용, (4) 재위탁에 해당하는 연동의 **식별 가능성**, (5) 일본 **표시·i18n·법적 고지** 확장 여지, (6) **국외 이전** 분기 시 APPI 전제와 주석이다.

## ISO 27001 코드 주석 (필수)
* **의무:** 보안·인증·권한·API·입력값·암호화·비밀·PII·감사 로그·Chronicle·에러 노출·속도 제한 등 **규정 준수와 직접 연결되는 기능**을 구현·수정할 때는, 해당 로직에 **ISO 27001 통제 번호(아래 각 절의 A.xx)와 이 문서의 절 번호(§1~§8)** 를 **주석으로 반드시 표기**한다. 통제를 **어떻게 충족하는지**는 **한국어·日本語·English 세 언어**로 각각 한 문장(의미 동등)을 적는다.
* **형식·예외:** 레이블 **KO / JA / EN**, 블록 주석 예시·매핑 표는 **SECURITY_GOVERNANCE.md §5**를 따른다. 순수 시각(레이아웃·색·타이포만) 변경은 생략 가능하다.

## 디자인 철학 (Classic Luxury)
* **최우선 가치:** 모든 UI(웹·모바일·공유 컴포넌트)는 **프리미엄 가치**를 가장 먼저 증명해야 한다. 기능·접근성·보안과 충돌하지 않는 범위에서 품질·완성도·신뢰감을 절충하지 않는다.
* **팔레트:** 배경은 **near-black 차콜**(`#0E0E0E` 등)로 통일해 캔버스를 고정한다. 강조·내비게이션·메탈릭 타이포에는 **챔페인 브라스(Champagne Brass)** — 채도를 낮춘 **밝은 황동** 톤(예: 악센트 `#DEB892`, `packages/config`·`globals.css` 토큰)을 절제 있게 쓴다. 본문·보조 텍스트는 **웜 화이트**로 대비를 유지한다.
* **재질(면) 표현:** CTA·primary 등 **금속 느낌의 면**은 단색 채우기나 원색 광택 대신, **다중 스톱 선형 그라데이션**과 **아주 얕은 하이라이트·음영 레이어**(예: `opus-surface-metallic`)로 **표면의 빛과 질감**만 드러낸다. 목적은 장식이 아니라 **재료가 받는 광의 방향성**을 읽게 하는 것이다. 스티커형 네온·과포화 레인보·플라스틱한 글로우는 지양한다.
* **무드:** **고전적 럭셔리(Classic Luxury)** — 절제된 장식, 넉넉한 여백, 정돈된 타이포그래피, 미세한 대비와 계조로 무게감을 낸다. 트렌디함보다 시간이 지나도 흐트러지지 않는 품격을 목표로 한다.
* **서비스 포지셔닝(상시 규칙):** OPUS는 NFT 기술을 적용하되, 이용자가 본 서비스를 **실물 투자·금융 자산 상품**으로 인식하게 하지 않는다. 카피·UI·메타데이터·정보구조는 **복제 불가능한 디지털 아트의 인증·소유·감상 경험**을 중심으로 구성하고, 수익·투자수단·가격상승 기대를 연상시키는 표현은 배제한다. 이 원칙은 신규 기능·페이지·마케팅 카피·앱 업데이트를 포함한 **향후 모든 작업에 기본값으로 적용**한다.
* **실행:** `packages/ui` 및 앱별 테마·토큰은 위 원칙에 맞춰 유지·확장하며, 보안·개인정보(PII) 표기 구역은 가독성과 신뢰를 해치지 않는 선에서 동일 철학을 적용한다.
* **타이포그래피(폰트):** 웹 서비스와 앱에 사용하는 폰트는 **저작권·라이선스 측면에서 사용이 자유롭거나 조건이 명확한 것**만 쓴다. 예: SIL Open Font License(OFL), Apache 2.0 등 허용 범위가 문서화된 오픈 폰트, 플랫폼 **시스템 UI 폰트**, 또는 **상용 라이선스·웹폰트 계약이 프로젝트에 근거로 남는** 경우. 라이선스 미확인·무단 상용 폰트 임베딩은 금지한다.
* **서비스명(브랜드):** 제품·UI 카피·앱·스토어·법적·마케팅 표기는 **OPUS**만 사용한다. 초기 기획 가칭(덴코마켓 등)은 코드·화면·문서에 넣지 않는다.
* **기획 문서(참고용):** 작품거래사이트.pdf 등 초안에서 뽑은 기능·일정·수익 가정 목록은 **참고만** 한다. 이미 반영된 항목, 보류·취소·변경될 수 있으며 구현 우선순위는 별도 합의로 조정한다.
* **시각 기준:** 웹·앱 레이아웃·밀도·카피 톤은 `apps/web/public/design-spec`(원본 PNG 및 PDF에서 추출한 시안 이미지 포함)과 **정합**을 맞춘다. 시안에 과거 가칭이 보여도 **구현 시 표기는 OPUS로 통일**하고, OPUS 토큰(**near-black 차콜·챔페인 브라스·웜 화이트**, 위 **재질 표현** 규칙)을 유지한다. **인장·공식 인증** 등은 `opus-seal` 계열로 소량 포인트만 쓴다.

## 고해상 작품 이미지 · 스마트폰 앱 (배포 정책)
* **원칙:** **구매자에게 제공하는 고해상도(또는 원본에 가까운) 감상용 래스터는 스마트폰 앱에서만** 제공한다. 웹(`apps/web`) 마켓플레이스는 공개·세션 한도 내 **저해상·워터마크 파생**만 유지한다(현재 `/api/catalog-image`의 `preview` / `vault` 등).
* **앱 제작 시:** 모바일 앱은 **인증된 구매자(Chronicle·에디션 소유 검증)** 만 고해상 자산 URL 또는 스트림을 받도록 API·CDN·DRM(필요 시) 경계를 설계한다. 웹과 동일 엔드포인트로 고해상을 내리지 않는다.
* **EN (for agents):** Ship **high-fidelity artwork pixels only inside the authenticated mobile app** for verified owners; keep the web on downsampled, watermarked derivatives unless product explicitly changes this policy.

## 1. 입력값 검증 및 살균 (Input Validation & Sanitization)
* **통제 항목:** ISO 27001 A.14.2.1
* **규칙:** 모든 클라이언트 입력값(Form, API Parameter, URL Query 등)은 '신뢰할 수 없는 데이터'로 간주한다.
* **실행:** SQL Injection, XSS, OS Command Injection 방지를 위해 반드시 프레임워크 제공 라이브러리를 사용하여 Sanitization을 수행하고, 화이트리스트 기반의 유효성 검사를 실시한다.

## 2. 인증 및 세션 관리 강화 (Strong Authentication & Session)
* **통제 항목:** ISO 27001 A.9.4.2
* **규칙:** 모든 인증은 다중 요소 인증(MFA) 확장이 가능한 구조로 설계하며, 세션 토큰은 안전하게 관리한다.
* **실행:** JWT 사용 시 서명 알고리즘을 강제하고 만료 시간을 최소화한다. 클라이언트 측 저장 시 `HttpOnly`, `Secure` 플래그를 필수로 적용한다.

## 3. 데이터 암호화 표준 준수 (Cryptographic Controls)
* **통제 항목:** ISO 27001 A.10.1.1
* **규칙:** 모든 민감 데이터는 전송 중(In-Transit) 및 저장 시(At-Rest) 암호화되어야 한다.
* **실행:** TLS 1.3 이상만 허용하며, 저장 시 AES-256 이상의 알고리즘을 사용한다. 암호화 키는 코드에 노출하지 않고 GCP Secret Manager를 통해서만 참조한다.

## 4. 최소 권한 원칙의 액세스 제어 (Least Privilege RBAC)
* **통제 항목:** ISO 27001 A.9.2.1
* **규칙:** 사용자 및 시스템 계정은 업무 수행에 필요한 최소한의 권한만 부여받는다.
* **실행:** 역할 기반 액세스 제어(RBAC)를 구현하고, 하드코딩된 관리자 계정을 절대 생성하지 않는다. API 엔드포인트는 항상 권한 체크 로직을 포함해야 한다.

## 5. 보안 에러 처리 및 감사 로그 (Secure Logging & Auditing)
* **통제 항목:** ISO 27001 A.12.4.1
* **규칙:** 에러 메시지를 통해 시스템 정보가 노출되지 않아야 하며, 모든 중요 행위는 감사 로그로 남긴다.
* **실행:** 사용자에게는 추상화된 에러 메시지만 노출하고, 상세 로그는 PII(개인정보)를 마스킹하여 별도의 보안 로그 서버에 기록한다. 특히 'The Chronicle'의 모든 변경 이력은 무결성이 보장되는 로그로 남긴다.

## 6. API 보안 및 가시성 제어 (API Security)
* **통제 항목:** ISO 27001 A.13.1.3
* **규칙:** 외부로 노출되는 모든 API는 속도 제한(Rate Limiting)과 인증이 적용되어야 한다.
* **실행:** API Gateway 패턴을 고려하여 설계하고, 불필요한 정보(Internal IP, Stack Trace 등)가 응답 헤더에 포함되지 않도록 설정한다.

## 7. 일본 개인정보보호법(APPI) 준수 (Privacy by Design)
* **통제 항목:** ISO 27001 A.18.1.4
* **규칙:** 개인정보(PII) 수집은 목적에 필요한 최소한으로 제한하며, 설계 단계부터 익명화를 고려한다.
* **실행:** 일본 APPI 가이드라인에 따라 사용자 동의 철회 및 데이터 삭제 기능을 구현하고, DB 설계 시 개인정보 테이블을 물리적/논리적으로 분리한다.

## 8. 의존성 및 공급망 보안 (Dependency Security)
* **통제 항목:** ISO 27001 A.15.1.1
* **규칙:** 모든 외부 라이브러리는 보안 취약점 점검을 통과한 최신 안정 버전만 사용한다.
* **실행:** `npm audit` 또는 `Snyk`과 같은 도구를 빌드 파이프라인에 통합하고, 알려진 취약점이 있는 패키지 발견 시 즉시 업데이트하거나 대체한다.

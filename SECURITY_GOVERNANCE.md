# 🛡️ OPUS Security Governance (ISO 27001 & Privacy)

본 문서는 OPUS의 보안 무결성을 위해 AI 에이전트가 코드를 짤 때 반드시 지켜야 할 기술적 상세 규칙을 정의한다.

### 0. 일본 기업 인수(M&A) 적합성 (설계 전제)
OPUS는 **향후 일본 법인 등에 사업·자산 양도가 이루어질 수 있음**을 전제로, 처음부터 **실사(Due Diligence)와 규제 대응에 유리한 형태**로 만든다. (법률 자문이 아니라 **엔지니어링·거버넌스 목표**다.)

- **추적 가능성:** 보안·개인정보 관련 로직에는 `CLAUDE.md`·본 문서 및 **ISO 27001 통제 주석(§5)** 으로 근거를 남기며, 설명은 **한국어·日本語·English** 삼어(§5)로 적어 인수측 기술·컴플라이언스 검토가 가능하게 한다.
- **데이터 경계:** PII·거래·Chronicle·감사 로그의 **역할과 저장 위치**를 모듈·스키마 단에서 분리해, 이전 시 범위·책임이 드러나게 한다. APPI상 **목적 외 이용·제3자 제공**이 코드 경로만으로도 구분되도록 설계한다.
- **라이선스·IP:** 폰트·라이브러리·API는 **허가 범위가 문서화 가능한 것**만 사용하고, 상용 구성요소는 계약서·인벤토리에 적을 수 있는 이름·버전으로 고정한다 (`CLAUDE.md` 타이포·§8 의존성).
- **외부 의존성:** 시크릿·결제·이메일 등 **재위탁(サブプロセッサ)** 에 해당하는 연동은 코드·환경변수·설정으로 식별 가능하게 두고, 장기적으로 **목록화(인벤토리)** 할 수 있는 구조를 유지한다.
- **일본 시장 연속성:** UI·카피·법적 고지(예: 특정商取引法に基づく表示、個人情報の取扱い)는 **i18n·정책 URL·문구 분리**로 확장 가능하게 두어, 인수 후 일본 운영 정책에 맞게 교체·보강하기 쉽게 한다.
- **국외 이전:** 개인정보의 국외 저장·처리가 생기면 **APPI 및 일본 관련 가이드라인**에 맞는 절차·동의·계약을 전제로 구현하며, 해당 분기에는 코드 주석에 **APPI·국외 이전** 근거를 **KO/JA/EN**으로 명시한다(§5).

### 1. 인증 및 권한 (Identity & Access)
- 모든 API는 **JWT 기반 인증**을 필수로 하며, `HttpOnly`, `Secure` 쿠키 정책을 사용한다.
- **RBAC(Role-Based Access Control):** '작가', '컬렉터', '관리자' 권한을 엄격히 분리하여 설계한다.

#### 1.1 계단식 신원·연령 확인 (step-up: 권한이 필요할 때 인증한다)
전 사용자에게 신분증을 요구하면 이탈이 커지므로, **최소 수집·단계적 강화**를 원칙으로 한다. (법률 자문이 아닌 **제품·엔지니어링 전제**이며, 출시 전 법무·규제 검토로 확정한다.)

| 구분 | 수단 | 비고 |
|------|------|------|
| **일반 이용자 (Buyer / 컬렉터)** | **SNS·OAuth 로그인** + **「만 18세 이상」자가 진술 체크박스** | 신분증 불요. 동의·연령 진술 시각은 감사용으로만 최소 보관 (`buyer_age_self_attested_at` 등). |
| **R-18 콘텐츠 구매·열람** | **신용카드 기반 연령·본인 확인** (결제/인증 PSP 등) | 일본에서는 **카드 보유·인증**이 성인 확인의 1차 수단으로 널리 쓰인다. **카드번호·CVV·신분증 이미지는 OPUS DB에 저장하지 않고**, 업체가 반환하는 **불투명 참조 ID·검증 시각**만 매핑한다 (`is_adult`, `credit_card_verification_ref`). |
| **판매자 (Seller / 작가)** | **전문 eKYC 솔루션** (신분증 + 얼굴 대조 등) | **원본 이미지·파일 경로는 DB에 두지 않는다.** 업체 처리 결과·상태·참조 ID만 저장 (`seller_ekyc_status`, `seller_ekyc_external_ref`). |

- **JA (要約):** 全員に身分証を求めず、**必要な権限のときだけ**段階的に強化する。買い手はSNS＋18歳以上チェック、R-18はカード認証、売り手はeKYC。画像パスやPANは当社DBに保存しない。
- **EN (summary):** Do not require ID from everyone; **step up when a permission requires it.** Buyers: OAuth + 18+ attestation. R-18: card-based age verification (common in Japan). Sellers: full eKYC via vendor. No PAN, CVV, or ID image paths in the OPUS database—vendor references only.

구현 시 `prisma/schema.prisma`의 필드명·주석과 본 절을 **동기화**하고, API·화면 분기에는 **ISO 27001 A.18.1.4 (§7) · CLAUDE.md §7** 추적 주석을 남긴다.

### 2. 데이터 보호 (Data Protection)
- **PII(개인정보):** 일본 APPI 규정에 따라 이름, 이메일 등은 DB 저장 시 암호화하거나 식별 불가능하게 처리한다.
- **The Chronicle:** 모든 소유권 변경 이력은 수정 불가능한(Immutable) 로그 형태로 설계하며, DB 수준에서 감사 추적(Audit Trail)이 가능해야 한다.

### 3. 통신 및 인프라 (Network & Infra)
- **Tailscale Only:** 모든 백엔드 관리 포트는 공인 IP가 아닌 Tailscale 사설망을 통해서만 접근하도록 설정한다.
- **Secret Management:** API 키, DB 비밀번호는 `.env`에 직접 박지 말고 GCP Secret Manager 연동을 전제로 작성한다.

### 4. 에러 및 예외 처리 (Error Handling)
- 시스템 내부 스택 트레이스(Stack Trace)가 외부 응답으로 유출되지 않도록 전역 에러 핸들러를 구현한다.
- 보안 관련 실패(로그인 실패 등)는 별도의 보안 로그 파일에 상세히 기록한다.

### 5. ISO 27001 코드 추적 주석 (필수)
구현 시 **해당 기능이 만족해야 할 통제**를 코드에 남겨, 감사·리뷰 시 ISO 27001과 `CLAUDE.md` 근거를 바로 찾을 수 있게 한다.

- **대상:** 인증·세션·권한(RBAC), API·입력 검증, 암호화·비밀·PII 처리, 감사 로그·Chronicle, 속도 제한·에러 응답, 의존성/빌드 보안 등 **보안·개인정보·규정 준수와 직결**되는 로직. 순수 UI 레이아웃·스타일만 바꾸는 변경은 생략 가능.
- **언어 (필수):** 규정 추적 주석의 **설명 문장은 한국어·日本語·English 세 가지 모두** 포함한다. **ISO 통제 번호(A.xx)·`CLAUDE.md` 절 표기**는 원문(라틴 문자) 그대로 한 번만 적어도 된다.
- **형식 (권장):** 블록 주석 권장. 첫 줄에 통제·절 참조, 이어 **KO / JA / EN** 각각 한 줄(의미 동등·간결).
  - `ISO 27001: A.9.2.1 · CLAUDE.md §4`
  - `KO: 역할 검증 후에만 입찰 API를 호출하도록 한다.`
  - `JA: ロール検証後にのみ入札APIを呼び出す。`
  - `EN: Calls the bid API only after role verification.`
  - 여러 통제: `ISO 27001: A.9.4.2, A.13.1.3 · CLAUDE.md §2, §6` 아래에 동일하게 KO/JA/EN 세 줄.
- **매핑 (요약):** 세부 실행은 `CLAUDE.md` 동일 번호 절을 따른다.
  - §1 입력 검증·살균 → **A.14.2.1**
  - §2 인증·세션 → **A.9.4.2**
  - §3 암호화 → **A.10.1.1**
  - §4 RBAC·최소 권한 → **A.9.2.1**
  - §5 로깅·감사 → **A.12.4.1**
  - §6 API 보안 → **A.13.1.3**
  - §7 APPI·프라이버시 → **A.18.1.4**
  - §8 의존성·공급망 → **A.15.1.1**
- **APPI/PII:** 일본 개인정보를 다루는 분기에는 통제 줄에 **APPI**를 명시하고, 설명은 **KO/JA/EN**으로 적는다. 예: `CLAUDE.md §7 (APPI)` 다음에 KO·JA·EN 각각 「수집 목적 최소화」에 해당하는 문장.
- 본 문서 §0~§4와 충돌하지 않게 작성하며, 상세 규칙은 `CLAUDE.md` 및 본 파일을 우선한다.

# OPUS 이중화 1단계 체크리스트 (현재 구조 즉시 적용)

목표: 대규모 재설계 없이 단일 장애점을 줄이고, 장애 시 복구 시간을 단축한다.

## 1) 컴퓨트/서비스 가용성

- [ ] EC2 재기동 시 자동 복구(systemd + docker compose) 검증
- [ ] 헬스체크 실패 시 자동 재시작 정책(`restart: unless-stopped`) 확인
- [ ] Caddy/Web 컨테이너 상태 모니터링 알림 연결
- [ ] 배포 스크립트 실패 시 자동 롤백 또는 수동 롤백 절차 문서화

## 2) 데이터/스토리지 보호

- [ ] `opus_storage` 일일 백업 크론 정상 동작 확인
- [ ] 백업 파일 보존기간/개수 정책 검토 및 주기적 점검
- [ ] 주 1회 복구 리허설(테스트 경로) 수행
- [ ] submissions/ownership-events 무결성 검사 스크립트 정기 실행

## 3) DB 가용성

- [ ] RDS 자동 백업/보존기간/스냅샷 정책 확인
- [ ] 장애 복구 시점(RPO)과 복구 시간(RTO) 목표 문서화
- [ ] Prisma migration 적용 절차(사전 백업 -> 배포 -> 검증) 고정
- [ ] 읽기 부하 증가 대비 읽기 복제본(Phase 2) 도입 조건 정의

## 4) 네트워크/접근 통제

- [ ] Security Group: 22/80/443 최소 허용 원칙 점검
- [ ] SSH 접근은 고정 키 + 허용 IP 제한 적용
- [ ] 운영자 계정/권한 분리 및 sudo 사용 감사 로그 점검
- [ ] 비상 접근 경로(키 분실/잠금) 복구 절차 문서화

## 5) 운영 자동 점검

- [ ] `scripts/ops-web-stability-check.sh`를 정기 실행(cron/CI)으로 연결
- [ ] `scripts/ops-release-e2e-check.sh`를 배포 후 자동 실행
- [ ] 실패 시 알림 채널(Slack/Email 등)로 즉시 통보
- [ ] 주간 운영 리포트(가용성, 에러율, 백업 성공률) 기록

## 6) 2단계 확장 기준(미리 정의)

- [ ] ALB + Auto Scaling(다중 AZ) 전환 트리거 조건 정의
- [ ] S3 원본 저장/CloudFront 서명 URL 전환 계획 수립
- [ ] RDS Multi-AZ 및 읽기 분리 도입 시나리오 문서화
- [ ] 재해복구(리전 단위) 목표와 비용 한도 합의

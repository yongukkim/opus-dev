# OPUS 결제 도입 전 체크리스트 (Japan Go-Live)

목표: 일본 서비스 기준으로 결제 도입 시 보안/법무/운영 누락 없이 출시 가능한 상태를 만든다.

## 1) 계정/정산/법무 준비

- [ ] 일본 법인/사업자 정보, 정산 계좌, 세금 처리 정책 확정
- [ ] 결제 대행사(PSP) 약관 검토 및 재위탁(서브프로세서) 목록 문서화
- [ ] 환불/취소/분쟁 대응 정책 문구(ko/ja/en) 확정
- [ ] 이용약관/개인정보처리방침 결제 섹션 업데이트
- [ ] APPI 대응(동의, 보관기간, 삭제/철회 경로) 점검

## 2) 기술 아키텍처 체크

- [ ] 결제 Webhook 수신 엔드포인트를 앱/API 레이어에 분리
- [ ] Webhook 서명 검증 실패 시 즉시 4xx 반환 및 이벤트 폐기
- [ ] 멱등키(idempotency key) 적용으로 중복 결제/중복 처리 방지
- [ ] 결제 상태 머신(`pending -> paid -> failed/cancelled/refunded`) 정의
- [ ] 결제 성공 이후 소장품 이력(Chronicle) 반영 트랜잭션 원자성 검증

## 3) 보안 통제 (ISO 27001 매핑)

- [ ] A.14.2.1 입력 검증: 금액/통화/주문ID/사용자ID 서버 재검증
- [ ] A.9.4.2 인증/세션: 결제 시작/조회 API 인증 및 세션 타임아웃 점검
- [ ] A.10.1.1 암호화: 결제 관련 비밀키는 Secret Manager만 사용
- [ ] A.12.4.1 감사로그: 결제 이벤트 추적ID, 상태변경, 오류를 마스킹 로깅
- [ ] A.13.1.3 API 보안: Webhook/IP 제한, rate limit, 내부 오류 비노출

## 4) 데이터 모델/운영

- [ ] `orders`, `payments`, `payment_events` 테이블 분리(PII 최소화)
- [ ] 결제 이벤트 원본(payload) 보관 정책과 마스킹 정책 확정
- [ ] 실패 재처리 배치(Dead Letter Queue 또는 재시도 큐) 준비
- [ ] 운영 대시보드(성공률/실패율/미처리 Webhook) 알림 연결
- [ ] 일일 정산 대사(PSP 리포트 vs DB 레코드) 수동 점검 절차 작성

## 5) QA / 출시 리허설

- [ ] 스테이징에서 카드 승인/실패/취소/환불 E2E 시나리오 완료
- [ ] 동시성 테스트(중복 클릭, 네트워크 재시도, Webhook 지연) 완료
- [ ] 롤백 시나리오(결제 성공, 소장품 반영 실패) 복구 절차 검증
- [ ] 보안 점검(권한 우회, 파라미터 변조, CSRF/Replay) 완료
- [ ] 출시 체크포인트(릴리즈 창구, 책임자, 비상연락망) 확인
- [ ] `docs/payment-webhook-smoke-runbook.md` 기준 webhook 스모크 테스트 정기 실행

## 6) 출시 당일 운영 체크

- [ ] 결제 기능 플래그 ON 이전 상태 점검(에러율, 큐 적체, DB 상태)
- [ ] 기능 플래그 단계적 오픈(내부 -> 제한 사용자 -> 전체)
- [ ] 1시간/24시간 모니터링 항목 담당자 배정
- [ ] 장애 시 즉시 플래그 OFF + 수동 주문 보호 프로세스 준비
